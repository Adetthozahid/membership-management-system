9. Risk Level
-------------
Overall risk level: Medium

High-risk structure areas:

1. Very large all-in-one files.
2. Scattered upload handling.
3. Duplicate or typo routes.
4. Missing test/spec folders.
5. Direct Prisma access inside large services.
6. Direct frontend API calls inside many pages/components.

Medium-risk structure areas:

1. Shared types centralized in one large file.
2. Missing repository layer.
3. Public-content module has no DTO folder.
4. Root contains many runtime artifacts.

Low-risk areas:

1. Monorepo separation.
2. NestJS module presence.
3. Prisma migration/seed placement.
4. Basic admin/member/public route separation.
5. Shared package existence.

10. Exact Recommended Refactor Order
------------------------------------
Recommended safe order:

1. Update .gitignore only for runtime artifacts.
   Add patterns such as:
   *.pid
   *.token
   *.upload.json
   *.tsbuildinfo
   uploads/
   apps/api/uploads/

2. Add test folder and script structure without changing app behavior.
   Suggested folders:
   apps/api/test/
   apps/web/__tests__/

3. Document legacy or typo routes.
   Specifically document:
   deashboad
   srithir-patha

4. Split packages/shared/src/types.ts into domain type files.
   Keep backwards-compatible exports so existing imports do not break.

5. Create frontend service folders.
   Move API calls behind service functions gradually.
   Do not change API behavior.

6. Make frontend route files thinner.
   Keep existing routes.
   Move feature logic into feature folders.

7. Extract apps/web/app/admin/website/page.tsx into:
   apps/web/features/admin/website/

8. Create backend common folders:
   apps/api/src/common/guards/
   apps/api/src/common/decorators/
   apps/api/src/common/upload/

9. Move guards/decorators carefully.
   Keep old imports as re-export wrappers until all imports are updated.

10. Centralize upload constants and helper service.
    Do not change /uploads/... URL paths.

11. Add repository layer feature-by-feature.
    Recommended order:
    membership-types
    renewals
    auth
    members
    public-content

12. Split public-content.service.ts.
    Suggested services:
    media.service.ts
    gallery.service.ts
    website-settings.service.ts
    blog.service.ts

13. Split members.service.ts.
    Suggested services:
    member-application.service.ts
    member-profile.service.ts
    member-payment.service.ts
    member-self.service.ts

14. Add DTO folder/files for public-content.

15. Only after tests and redirects are confirmed, clean up typo/legacy route folders.

Final Note
----------
The current structure is workable and not broken.
The main architectural concern is not separation between frontend/backend; that part is good.
The main concern is growth pressure: large files, scattered API/upload logic, missing repository layer, centralized shared types, and missing tests.

Recommended approach:
Avoid a big-bang refactor.
Make small, reversible, compatibility-preserving structure improvements first.
