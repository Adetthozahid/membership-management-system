MASTER PROMPT
You are a senior full-stack software engineer and product architect. Build a complete production-grade Membership Management System for a single organization such as an association, alumni group, community, club, chamber, or professional body.

This is NOT a SaaS product. Do not build multi-tenant subscription, multiple organization billing, package management, subdomain management, or SaaS admin features. The system is for one organization only, but it must be configurable enough for different association-style requirements.

The system must be fully functional. Do not create a demo-only system. Do not use fake data as the core implementation. Do not leave important workflows as placeholders. Every module you implement must have working backend, database persistence, validation, authentication/authorization, frontend UI, and practical error handling.

Use this technology stack unless the project owner explicitly changes it:

- Frontend: Next.js with TypeScript
- Backend: NestJS with TypeScript
- Database: PostgreSQL
- ORM: Prisma
- UI: Tailwind CSS with shadcn/ui-style components
- Auth: JWT access token plus refresh token
- File upload: local storage first, designed so S3-compatible storage can be added later
- PDF/ID generation: server-side HTML-to-PDF or image/PDF generation
- QR code: server-generated QR linked to public verification page
- Deployment: Docker Compose on VPS with Nginx/reverse proxy support

Recommended monorepo structure:

membership-management-system/
  apps/
    web/                 # Next.js frontend: public website, member panel, admin panel
    api/                 # NestJS backend API
  packages/
    shared/              # shared types, constants, validation helpers
  docker-compose.yml
  README.md

Core requirement:
Build a complete Membership Management System with public website, admin panel, member dashboard, dynamic form builder, registration approval workflow, membership type and renewal rules, committee management, notice/event modules, payment/chanda/donation tracking, ID card with QR verification, gallery, reports, role permission, audit logs, and future-ready structure for election and accounts modules.

General engineering rules:

1. Use TypeScript everywhere.
2. Use PostgreSQL relational schema with Prisma migrations.
3. Use clean module-based architecture in NestJS.
4. Use DTO validation, guards, policies/permissions, and consistent API responses.
5. Use pagination, filtering, sorting, and search for admin lists.
6. Use real authentication and authorization.
7. Use secure password hashing.
8. Use audit logging for important admin actions.
9. Use file validation for uploaded images/documents.
10. Use environment variables for secrets.
11. Include seed data only for initial roles, permissions, settings, and optional development testing, not as a substitute for real features.
12. Include README setup instructions.
13. Include Docker Compose for local and server deployment.
14. Build responsive UI for desktop and mobile.
15. Never hardcode organization-specific registration fields; use the dynamic form builder.

Notes For Developer

  • Registration fields hardcode করা যাবে না। শুধু core identity fields relational table-এ থাকবে, বাকিগুলো form builder-driven হবে।
  • Expired/non-renewed member public directory থেকে hide হতে পারে, কিন্তু admin panel থেকে কখনো হারাবে না।
  • Committee term শেষ হলে delete নয়, archive করতে হবে।
  • Member ID, receipt number, QR token এগুলো predictable বা insecure করা যাবে না।
  • Admin যেন sensitive private form fields public website-এ accidentally expose করতে না পারে।
  • Election module করলে voting secrecy খুব কঠোরভাবে maintain করতে হবে।
